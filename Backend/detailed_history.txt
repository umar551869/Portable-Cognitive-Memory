--- LOG SEGMENT 5: FINAL YEAR PROJECT - PORTABLE COGNITIVE GRAPH ---

The Portable Cognitive Graph (PCG) is Umar's capstone final year project. It is a system designed to create a persistent, queryable knowledge graph from unstructured text logs, code interactions, and system activities. The core idea is that every piece of information a user encounters or produces can be semantically extracted, decomposed into entities and relationships, and stored as a graph structure that grows over time — effectively becoming an external "second brain."

The PCG architecture consists of several key subsystems:

The Ingestion Pipeline handles raw text input from multiple sources. It uses a chunking strategy that breaks large documents into overlapping segments of approximately 800 tokens with 200-token overlap, preserving context boundaries. Each chunk is processed independently through the entity extraction and relationship extraction stages.

The Entity Resolution system uses UUID5-based deterministic node ID generation, incorporating user_id to ensure per-user graph isolation. When a new entity is extracted, the resolver checks for existing nodes with similar canonical names, merging duplicates and updating metadata. Entity types include "concept" (abstract ideas), "process" (workflows and procedures), "mechanism" (technical implementations), and "decision" (choices and tradeoffs).

The Embedding System generates vector representations using Ollama's local embedding models. Each chunk, node, and edge receives a dense vector embedding that enables semantic similarity search. These embeddings are stored alongside the graph entities in the SQLite database, with plans to migrate to PostgreSQL with pgvector for production scale.

The LLM Provider system uses a fallback pattern: it first attempts to use the local Ollama model (llama3.1:8b) for entity and relationship extraction, falling back to Gemini API if the local model is unavailable. The prompts are carefully designed to extract structured JSON containing entities with names, types, descriptions, and aliases, as well as relationships with source entities, target entities, relation types, weights, and evidence.

The Graph Storage layer uses SQLAlchemy async with SQLite (aiosqlite driver). The schema includes tables for users, raw_logs, chunks, nodes, edges, and embeddings. Nodes have fields for canonical_name, display_name, type, description, aliases (JSON), weight, and metadata (JSON). Edges have source_id, target_id, relation, weight, evidence, and metadata.

The API Layer is built with FastAPI, providing RESTful endpoints for authentication (/auth/login, /auth/register), graph retrieval (/graph), semantic recall (/recall), and log ingestion (/ingest). JWT tokens are used for stateless authentication. CORS middleware is configured to allow the React frontend to communicate with the backend.

The Frontend is a React application built with Vite, featuring a 3D force-directed graph visualization using react-force-graph-3d and Three.js. Nodes are rendered as glowing spheres with colors based on their type, and edges are rendered as particle-animated links. The UI includes a login system, a semantic search bar, a node detail panel showing descriptions and relationships, and an interactive 3D scene where users can fly through their knowledge graph.

--- LOG SEGMENT 7: SYSTEM INTEGRATION & GRAPHIFY ---

Umar integrated the graphify framework into the PCG system to enable deterministic structural extraction from source code. Graphify uses tree-sitter, a parser generator tool and incremental parsing library, to build concrete syntax trees for source code files. Unlike LLM-based extraction which is probabilistic and slow, tree-sitter provides exact, deterministic AST (Abstract Syntax Tree) parsing at near-instantaneous speed.

The integration works through a StructuralExtractor class that maps file extensions to graphify's LanguageConfig objects. When a code file is ingested, the extractor first runs the AST pass to identify classes, functions, imports, and their relationships (contains, inherits, imports_from, calls). These structural nodes are then merged into the PCG graph alongside the LLM-extracted semantic nodes, creating a two-layer knowledge representation: structural (exact, deterministic) and semantic (probabilistic, contextual).

Graphify supports 15+ programming languages including Python, JavaScript, TypeScript, Java, C, C++, Ruby, C#, Kotlin, Scala, PHP, Lua, Swift, Go, and Rust. Each language has a dedicated LanguageConfig that defines class types, function types, import types, call types, and custom import handlers for resolving module dependencies.

The Leiden community detection algorithm, used by graphify for graph clustering, partitions the knowledge graph into communities of densely connected nodes. This reveals hidden structure in the graph — clusters of related concepts that might not be obvious from individual edges. The algorithm optimizes the Constant Potts Model (CPM) quality function and produces a hierarchical community structure.

--- LOG SEGMENT 8: SYSTEM SENSOR & COGNITIVE MONITORING ---

The System Sensor is a background service that monitors the user's PowerShell command history file at %APPDATA%\Microsoft\Windows\PowerShell\PSReadLine\ConsoleHost_history.txt. It uses an async polling loop that checks for new content every 5 seconds by comparing the current file size against the last known size. When new commands are detected, they are automatically ingested into the PCG through the standard processing pipeline.

This creates a passive learning system where the PCG continuously absorbs the user's technical workflow — every command they run, every tool they use, every directory they navigate. Over time, this builds a detailed map of the user's technical habits, preferred tools, common workflows, and project-switching patterns. The sensor is power-aware and designed to minimize system impact.

Future sensor expansions include browser history monitoring for tracking research patterns and learning resources, IDE activity tracking for understanding code editing patterns, Git commit history analysis for project evolution mapping, and clipboard monitoring for capturing frequently referenced information.

--- LOG SEGMENT 12: PROJECT ARCHITECTURE DECISIONS ---

Several key architectural decisions define the PCG system:

Decision 1: SQLite over PostgreSQL for initial development. SQLite was chosen for zero-configuration setup, file-based portability (the entire graph is a single pcg.db file), and sufficient performance for single-user development. The plan is to migrate to PostgreSQL with pgvector when the graph exceeds development scale, enabling concurrent access and efficient vector similarity search.

Decision 2: Local-first LLM processing with cloud fallback. The LocalProvider uses Ollama for on-device inference, eliminating API costs and maintaining data privacy. The GeminiProvider serves as a fallback when Ollama is unavailable or when higher-quality extraction is needed. This dual-provider pattern allows seamless switching based on availability and quality requirements.

Decision 3: Deterministic AST extraction before probabilistic LLM extraction. By running graphify's tree-sitter pass first, the system captures exact structural relationships (imports, inheritance, function calls) without LLM latency or token costs. The LLM then adds the semantic layer — understanding the "why" and "how" behind code structures. This two-pass approach provides both precision and depth.

Decision 4: Per-user graph isolation using UUID5 namespacing. Node IDs are generated deterministically from the combination of user_id and canonical entity name, ensuring that two users who encounter the same concept get separate, isolated graph nodes. This enables multi-tenant deployment without data leakage between users.

Decision 5: Power-aware background processing. The file monitor checks system battery state before queuing intensive ingestion tasks, deferring processing when on battery power. This respects the user's hardware constraints and prevents the cognitive graph from draining laptop batteries during mobile use.

Decision 6: Overlapping chunk windows for context preservation. The 800-token chunks with 200-token overlap ensure that entities mentioned near chunk boundaries are captured in at least one chunk's context window. This reduces information loss at segment boundaries and improves entity extraction quality.
